# SACT Intelligent Scheduling Optimisation System

**Velindre Cancer Centre | MSc Data Science Dissertation | Cardiff University**

An ML-enhanced scheduling optimisation system for Systemic Anti-Cancer Therapy (SACT). Combines 16 machine learning models — including an **inverse-RL preference learner** (§1.4), a **decision-focused calibration head** (§2.1 Smart Predict-then-Optimise), a **risk-adaptive conformal α policy** (§2.2), a **Temporal Fusion Transformer** replacing the three-model stack with a single attention-based joint model (§2.3), and a **5-% RCT randomisation layer** whose unbiased ATE is used as a Bayesian prior on the observational DML estimator (§2.4) — with multi-objective constraint optimisation (Google OR-Tools CP-SAT) to improve cancer treatment scheduling. Built to operate at Velindre Cancer Centre and its four outreach sites across South Wales.

> **Research context:** No existing NHS system (ChemoCare, ChemoSchedule, iQemo, CEPAS) applies machine learning to SACT scheduling. This system fills that gap with SACT v4.0 data standard compliance (NHS England, effective April 2026).

## 4. Data Channels

### Channel 2: Real Hospital Data (production)

Place real hospital data in `datasets/real_data/`. The system auto-detects this folder **both at startup and at runtime** — drop files while the server is running and it switches without a restart.

#### Auto-switch timeline (runtime)

A background "Channel 2 watcher" daemon polls `datasets/real_data/` every 60 s with two correctness guards:

| Time | Event |
|---|---|
| 0 s | Files arrive on disk |
| ≤ 60 s | Watcher detects new/updated files |
| 60–120 s | **Stability gate** — watcher waits one more polling cycle confirming `mtime` is unchanged (prevents racing with an in-flight write) |
| ≈ 2 min | Active channel flips from `synthetic` (Ch1) to `real` (Ch2); `initialize_data()` reloads patients + historical appointments |
| 2 – 5 min | All 12 ML models retrain on real outcomes in a background thread — no downtime, requests continue to be served |
| 5 min onwards | Every prediction, optimisation, fairness audit, and clinician override runs against models trained on real Velindre data |

**Monitor live:** `GET /api/data/channel2-watcher` returns current state (`active_channel`, `real_data_ready`, `real_data_missing`, `last_rejection_reason`, `last_check`, `last_switch`).

**Log you will see on successful promotion:**
```
Channel 2 watcher: real_data/patients.xlsx detected — auto-promoting from synthetic (Ch1) to real hospital data (Ch2)
Channel 2 watcher: switch complete — models retraining.
```

#### Correctness guards

- **Required files.** Both `patients.xlsx` AND `historical_appointments.xlsx` must be present. Dropping only one is rejected with `Channel 2 watcher: incomplete drop — missing required files: …` and the system stays on Channel 1.
- **Stability gate.** A file whose `mtime` is still changing between polls (export still writing) is not promoted until two consecutive polls show the same `mtime`.
- **No auto-downgrade.** If the real files are removed while Ch2 is active, the system does *not* revert to synthetic automatically — it logs a warning and keeps serving Ch2 until a manual `POST /api/data/source {"channel":"synthetic"}` is issued. Prevents silent demotion from an accidental file deletion.

#### How to convert real data to the required format

**If your system exports SACT v4.0 CSV (NHS standard):**

The `SACTv4DataAdapter` in `data/sact_v4_schema.py` automatically maps real NHS field names to the internal format. The following column names are recognised directly without renaming:

| Real SACT v4.0 field | Internal field |
|----------------------|----------------|
| `NHSNumber` | `NHS_Number` |
| `LocalPatientIdentifier` | `Local_Patient_Identifier` |
| `FamilyName` | `Person_Family_Name` |
| `DateOfBirth` | `Person_Birth_Date` |
| `Gender` | `Person_Stated_Gender_Code` |
| `DiagnosisCode` | `Primary_Diagnosis_ICD10` |
| `Regimen` | `Regimen_Code` |
| `TreatmentIntent` | `Intent_Of_Treatment` |
| `RegimenStartDate` | `Start_Date_Of_Regimen` |
| `Height` | `Height_At_Start` |
| `Weight` | `Weight_At_Start` |

Simply place the CSV in `datasets/nhs_open_data/sact_v4/` and the adapter runs automatically.

**If your system exports a different format (e.g., ChemoCare export, local CSV):**

1. Open your export in Excel or pandas
2. Rename columns to match the required column names in the table above
3. Save as `.xlsx` and place in `datasets/real_data/`
4. Restart the server — auto-detection will load and validate the data

**If you are using ChemoCare:**

ChemoCare typically exports in a proprietary format. Ask your system administrator for a "patient demographics export" and a "treatment history export". Map:
- Patient ID → `Local_Patient_Identifier`
- Date of birth → `Person_Birth_Date` (reformat to YYYY-MM-DD)
- Treatment attended/not attended flag → `Attended_Status` (Yes/No/Cancelled)
- Protocol → `Regimen_Code`
- Clinic site → `Site_Code` (map to WC/PCH/RGH/POW/CWM)

#### Step-by-step: loading real data for the first time

```
1. Export patient list and appointment history from your clinical system
2. Rename/reformat columns to match the specifications above
3. Save as patients.xlsx and historical_appointments.xlsx
4. Copy both files to: datasets/real_data/
5. Restart the server:  python flask_app.py
6. Watch startup logs — look for:
       [DATA] Real hospital data detected — all models retrained
       [ML] No-show ensemble trained: 1 models, AUC ~0.XX
       [ML] Duration ensemble trained, MAE ~XX min
7. Open http://localhost:1421 and verify patient count in Patients tab
8. Run a fairness audit (Optimise tab → Fairness Audit) to confirm
   demographic distributions look plausible
```

#### Data validation on load

The system runs `SACTv4Validator.validate()` against loaded files and assigns a grade A–F:

| Grade | Score | Meaning |
|-------|-------|---------|
| A | 90–100 | Full SACT v4.0 compliance — all ML features available |
| B | 75–89 | Minor gaps — most features available |
| C | 60–74 | Moderate gaps — some features estimated |
| D | 45–59 | Significant gaps — baseline models only |
| F | < 45 | Too many missing fields — synthetic fallback activated |

Validation report visible at: `POST /api/data/sact-v4/validate` or ML Models tab → Data Validation panel.

### Channel 3: NHS Open Data (auto-downloaded)

The ingestion engine runs every 24 hours in a background thread. No manual action required. Data is downloaded to `datasets/nhs_open_data/`.

| Source | Directory | Format | Frequency | Feeds |
|--------|-----------|--------|-----------|-------|
| **Cancer Waiting Times** | `cancer_waiting_times/` | CSV | Monthly (1st) | No-show baselines, waiting-time targets → Level 1 recalibration |
| **⚠ NDRS Cancer Data Hub** | `ndrs_cancer_data_hub/` | HTML poll + CSV | Monthly (from Jul 2026) | **CWT successor** — activates automatically when CWT decommissions |
| **NHSBSA SCMD-IP** | `prescribing/` | JSON catalogue | Monthly (20th) | Drug prescribing context |
| **SACT Activity Dashboard** | `sact_activity/` | JSON metadata | Annually | Regimen distribution weights → Level 2 recalibration |
| **SACT v4.0 Patient Data** | `sact_v4/` | CSV (manual) | Monthly from Aug 2026 | Level 3 full retrain trigger |

**CWT → NDRS transition (June 2026):**

The NHS England Cancer Waiting Times system decommissions in June 2026. The system handles this automatically:
- On 1 July 2026 the `cancer_waiting_times` source auto-disables
- The `ndrs_cancer_data_hub` source activates and begins polling:
  - [NDRS Cancer Data Hub](https://digital.nhs.uk/ndrs/data/data-outputs/cancer-data-hub) — aggregate cancer outputs
  - [Faster Diagnosis Standard (FDS)](https://www.england.nhs.uk/statistics/statistical-work-areas/cancer-waiting-times/faster-diagnosis-standard/) — 28-day referral-to-diagnosis statistics
- Any discovered CSV/XLSX download links are attempted automatically
- A manifest JSON in `datasets/nhs_open_data/ndrs_cancer_data_hub/` lists all discovered links with instructions if manual download is needed

**Manual trigger:**

```http
POST /api/data/nhs/check-updates
```

Or from the ML Models tab → NHS Data Pipeline panel → "Check for Updates" button.

| Badge | Colours | What it tells you |

Hover for a tooltip that explains the current state — on Channel 1 it reports whether Channel 2 files are missing, already pending the stability gate, or simply absent; on Channel 2 it confirms the real hospital data is live. Channel 3 (NHS open data) is always running in the background for recalibration and is not a "primary" choice, so it does not appear on the badge.

### Tab 1: Schedule

The main scheduling view — a Gantt chart showing all appointments for a selected site and date.

**Controls:**
- **Site selector** — WC (Whitchurch), PCH (Prince Charles), RGH (Royal Glamorgan), POW (Princess of Wales), CWM (Cwmbran Mobile)

| Border colour | No-show probability | Recommended action |
|---------------|--------------------|--------------------|
| No border | < 10% | None |
| Yellow | 10–25% | Send reminder (SMS/letter) |
| Orange | 25–40% | Phone confirmation call |
| Red | > 40% | Consider double-booking this slot |

### Tab 2: Squeeze-In (Urgent Patients)

Add urgent patients to an already-published schedule without disrupting other appointments.

**Workflow:**
1. Enter patient details (name, NHS Number / Trust ID, regimen, priority).
2. Click **Preview Best Slots** — the engine scores every gap across every chair and returns up to 10 candidates. Results are displayed in a **scrollable list** ordered **chronologically** (morning → afternoon) so the page reads left-to-right as the day's timeline. Score is printed on each row.
3. Either:
   - Click the green **Insert here** button next to a specific row to pin the urgent patient into that exact chair/time (operator-selected manual insert), or
   - Click **Insert Now** at the top to let the engine pick the best-scoring slot automatically.

**Two-stage internal pipeline:**
- Stage 1 — score-ranked filter picks the top-10 candidates using gap size, robustness (min-slack before/after), CVaR worst-case, chair crowding, and a mild mid-morning preference.
- Stage 2 — survivors are sorted chronologically before being returned to the UI. Score ordering is preserved within ties.

**Patient ID enforcement (channel-aware):**

| Active channel | Empty / `PREVIEW` / `URGENT_*` IDs |
|---|---|
| Ch1 synthetic | Auto-generated silently: `URGENT_<timestamp>` — convenient for demos and staff training. |
| Ch2 real hospital | **Rejected at the server boundary** (`HTTP 400`). The frontend prompts the operator for a real NHS Number or Trust Local ID before firing the request. Prevents orphan appointments in the live schedule. |

**High No-Show Slot list (right panel):**
All upcoming appointments whose predicted no-show probability exceeds the low-risk threshold, diversified so no 30-minute bucket contributes more than two rows, then sorted **chronologically** across the day. Use this to scan the day for proactive phone calls or safe double-booking opportunities.

**Double-booking confidence:**
- **HIGH**: existing patient no-show probability > 50%, appointment > 2 days away
- **MEDIUM**: probability 30–50%, or < 2 days
- **LOW**: probability < 30% — not recommended for double-booking

### Tab 3: Optimise

Run the CP-SAT multi-objective optimisation engine.

**Single optimisation run:**
1. Select date and site
2. Choose an optimisation profile:
   - **Balanced** — equal weight across all objectives (default)
   - **Max Throughput** — maximise chairs filled; prioritises utilisation
   - **Patient First** — minimise waiting time; prioritises patient comfort
   - **Risk Averse** — minimise no-show exposure; maximises buffer
   - **Robust** — maximise schedule slack; best for unpredictable days
3. Click **Run Optimisation** (solve time up to 5 minutes for complex days)
4. Review the solution: appointments assigned, utilisation %, unscheduled patients (if any)

**Pareto Frontier:**
Generates 5 trade-off solutions simultaneously, one per profile. Displays a table comparing all 5 on all 6 objectives. Use this to understand the cost of each trade-off before committing.

**Fairness Audit:**
Checks the proposed schedule against the Equality Act 2010 protected characteristics:
- Age groups: < 40, 40–60, 60–75, > 75
- Gender: Male / Female / Not stated
- Distance: Near (< 10 km), Medium (10–30 km), Long (> 30 km), Remote (> 50 km)

Reports:
- Demographic parity difference (< 0.1 = good, > 0.2 = intervention needed)
- Disparate impact ratio (> 0.8 = acceptable, < 0.8 = potential adverse impact)

### Tab 8: ML Models

Full machine learning control panel.

**Model status cards (12 cards):**
Each card shows:
- Model name and purpose
- Training status (trained / untrained / stale)
- Last trained date and dataset size
- Key performance metric (AUC, MAE, C-index, etc.)
- **Run** button to generate a single prediction

**Individual prediction tools:**
Each model has its own input form. For example:
- **No-Show Ensemble**: enter patient age, regimen, distance, day of week, weather severity → returns probability
- **Duration Ensemble**: enter regimen, cycle number, performance status → returns predicted duration ± confidence interval
- **Conformal Prediction**: same inputs → returns guaranteed-coverage interval (e.g., 90% interval: 85–115 min)
- **MC Dropout**: same inputs → returns mean prediction + epistemic/aleatoric uncertainty split

**Causal analysis tools:**
- **Causal DAG**: displays the directed acyclic graph of causal relationships between variables
- **Counterfactual analysis**: "what if this patient had taken a different route?" — estimates causal effect
- **IV Analysis**: instrumental variable estimate of distance effect on no-show (using postcode district as instrument)
- **DML**: Double Machine Learning estimate of reminder intervention effect

**Optimisation weights editor:**
Adjust the 6 CP-SAT objective weights in real time (must sum to 1.0). Changes apply to the next optimisation run.

**NHS Data Pipeline:**
- Shows last download date for each of the 5 NHS data sources
- CWT → NDRS CDH transition status and countdown
- SACT v4.0 rollout phase and local file count
- "Check for Updates" button triggers immediate re-check of all sources

**Recalibration controls (4 levels):**

## 6. Machine Learning Models

The system uses 12 distinct machine learning models housed in the `ml/` directory. All 12 are trained on startup (or on recalibration trigger).

### Feature Engineering Pipeline

All models share a common 92-feature input space, derived from 22 raw input fields:

| Feature group | Input fields | Output features | Examples |
|--------------|-------------|-----------------|---------|
| Patient history | 8 | ~15 | `prev_noshow_rate`, `is_new_patient`, `days_since_last_visit` |
| Temporal | 6 | ~25 | `day_of_week`, `month`, `season`, `booking_lead_days` |
| Geographic | 3 | ~15 | `distance_km`, `travel_time_min`, `is_local` (< 10 km) |
| Treatment | 8 | ~18 | `expected_duration_min`, `is_immunotherapy`, `is_first_cycle` |
| External | 3 | ~9 | `weather_severity`, `traffic_severity`, `combined_external_severity` |

### Model 1: No-Show Ensemble (`ml/noshow_model.py`)

**Purpose:** Predict the probability that a specific patient will not attend their scheduled appointment.

**Method:** Stacked ensemble of three base learners:
- Random Forest (100 trees, `max_depth=10`)
- Gradient Boosting (`n_estimators=200, learning_rate=0.05`)
- XGBoost (`n_estimators=200, max_depth=6, subsample=0.8`)

Meta-learner: Logistic Regression trained on out-of-fold predictions from the three base learners (5-fold cross-validation stacking).

**Training:** Requires `Attended_Status` column in historical appointments. Binary target: 1 = no-show, 0 = attended.

**Output:** Probability 0–1. Thresholds:
- < 0.10 → Low (no action)
- 0.10–0.25 → Medium (send reminder)
- 0.25–0.40 → High (phone confirmation)
- > 0.40 → Very High (consider double-booking)

**Performance on synthetic data:** AUC 0.635 (stacked); training-set AUC 0.82 (single RF, not stacked).

**Key drivers (SHAP):** Historical no-show rate, booking lead time, travel distance, day of week, weather severity, treatment cycle number.

---

### Model 2: Duration Ensemble (`ml/duration_model.py`)

**Purpose:** Predict how long a treatment appointment will actually take.

**Method:** Stacked ensemble (same architecture as Model 1) for regression:
- Random Forest Regressor
- Gradient Boosting Regressor
- XGBoost Regressor

Meta-learner: Ridge Regression.

**Training:** Requires `Actual_Duration` column in historical appointments (attended appointments only; cancelled/no-show rows excluded).

**Output:** Predicted duration in minutes, ± confidence interval (via quantile regression).

**Performance:** MAE approximately 11 minutes on synthetic data.

**Key drivers:** Regimen code (inherent complexity), cycle number (first cycle takes longer), performance status (PS 2–3 requires more nursing support), has comorbidities, IV access difficulty.

---

### Model 3: Sequence Model (`ml/sequence_model.py`)

**Purpose:** Learn patient treatment trajectories — how a patient's no-show risk and duration evolve over successive treatment cycles.

**Method:** GRU (Gated Recurrent Unit) recurrent neural network implemented in PyTorch:
- Input: sequence of up to 10 previous appointments per patient
- GRU layer: 64 hidden units, single layer
- Output: predicted no-show probability and duration for the next appointment

**Training:** Requires patients with at least 2 historical appointments. Sequences padded to equal length.

**Output:** Trajectory-aware no-show probability and duration that accounts for trend (improving or deteriorating attendance over the treatment course).

**Use case:** Particularly useful for detecting when a patient's attendance pattern is deteriorating mid-treatment — earlier warning than population-mean estimates.

---

### Model 4: Survival Analysis (`ml/survival_model.py`)

**Purpose:** Model the time-to-event distribution for treatment delays, interruptions, and regimen modifications.

**Method:** Cox Proportional Hazards model via `lifelines`:
- Covariates: age, performance status, regimen, comorbidity count, toxicity grade (if available)
- Event: treatment delay > 7 days, dose modification, or treatment discontinuation
- Time: days from treatment start

**Output:**
- Hazard ratio per covariate (e.g., each unit increase in toxicity grade → 1.8× hazard)
- Survival curve for a specific patient: probability of remaining on schedule at day t
- Median time-to-modification estimate

**Use case:** Risk-stratify patients at the point of scheduling — patients with high cumulative hazard scores should be given priority slots with more buffer time.

---

### Model 5: Uplift Modelling (`ml/uplift_model.py`)

**Purpose:** Estimate the causal effect of an intervention (e.g., sending a reminder call) on attendance — not just whether reminded patients attend, but whether the reminder *caused* them to attend.

**Method:** Meta-learner framework with two approaches:
- **S-Learner**: single model with treatment indicator as a feature (XGBoost)
- **T-Learner**: two separate models — one for treated patients, one for control (Random Forest per group)

The Individual Treatment Effect (ITE) = `prediction(treated) − prediction(control)` for each patient.

**Output:** Per-patient uplift score:
- **Positive uplift** (> 0.05): reminder likely to increase attendance — high value target
- **Persuadable zone** (0.01–0.05): borderline — reminder may help
- **Negative uplift** (< 0): patient unlikely to respond to reminder — save resource

**Use case:** Prioritise which patients receive phone confirmation calls rather than calling all high-risk patients indiscriminately.

---

### Model 6: Multi-Task Learning (`ml/multitask_model.py`)

**Purpose:** Simultaneously predict no-show probability and treatment duration using a shared neural network — beneficial when the two tasks share underlying drivers.

**Method:** PyTorch shared-layer network:
- Shared encoder: `[92 → 128 → 64]` with ReLU and dropout (p=0.3)
- No-show head: `[64 → 32 → 1]` sigmoid output
- Duration head: `[64 → 32 → 1]` linear output
- Loss: `L = α × BCE_noshow + (1−α) × MSE_duration`, α = 0.5

**Training:** Both targets trained jointly. Dropout acts as regularisation; the shared representation forces the two tasks to inform each other.

**Output:** Joint prediction tuple `(noshow_probability, duration_minutes)` with a single forward pass — used when both are needed simultaneously (e.g., real-time schedule gap filling).

---

### Model 7: Quantile Regression Forest (`ml/quantile_forest.py`)

**Purpose:** Provide distribution-free prediction intervals for both no-show probability and treatment duration without parametric assumptions.

**Method:** 100-tree Random Forest trained as a quantile forest:
- Stores all training outcomes in each leaf node (not just the mean)
- At prediction time, draws from the empirical conditional distribution
- Returns `(q10, q50, q90)` — lower bound, median, upper bound for any desired coverage

**Two instances:**
- `qrf_duration`: predicts duration quantiles (uses attended appointments only)
- `qrf_noshow`: predicts no-show probability quantiles

**Output:** E.g., "80% chance the appointment will take between 82 and 127 minutes." The interval width reflects genuine uncertainty — narrow intervals indicate predictable cases; wide intervals flag high-variance patients.

**Use case:** Drive buffer allocation — the optimiser uses the 90th percentile duration prediction when computing schedule slack under the `robust` or `risk_averse` profile.

---

### Model 8: Hierarchical Bayesian Model (`ml/hierarchical_model.py`)

**Purpose:** Estimate patient-level random effects — some patients are inherently more punctual or more variable than average regardless of covariates.

**Method:** PyMC hierarchical model with NUTS sampler:
- Population-level priors: `μ_noshow ~ Normal(0.13, 0.05)` (population no-show rate)
- Patient-level random effects: `patient_effect_i ~ Normal(0, σ_patient)` (partially pooled)
- Likelihood: `Attended_i ~ Bernoulli(logistic(μ_noshow + patient_effect_i + β·x_i))`
- MCMC: 1,000 tuning + 2,000 draw samples, 2 chains

**Output:**
- Per-patient baseline risk (adjusted for history even with few data points)
- Full posterior distributions with credible intervals (not just point estimates)
- Shrinkage towards population mean for new patients (cold-start handling)

**Use case:** New patients (< 3 previous appointments) get the population mean with uncertainty. Patients with many appointments get a personalised estimate. The partial pooling prevents over-fitting to noisy individual histories.

---

### Model 9: Causal DAG + IV + DML (`ml/causal_model.py`)

**Purpose:** Move beyond correlation to causal understanding — estimate the true causal effect of distance, weather, reminder calls, etc. on attendance, accounting for confounding.

This module contains three causal methods:

**9a. Causal DAG (Directed Acyclic Graph):**
- Encodes the assumed causal structure: Weather → Attendance; Distance → Travel Time → Attendance; Reminder → Attendance; Performance Status → Attendance
- Uses `do`-calculus (Pearl 2009) to identify valid adjustment sets
- Allows counterfactual queries: "what would this patient's attendance probability have been if the weather had been fine?"

**9b. Instrumental Variables (IV — 2SLS):**
- Problem: Distance confounds attendance (farther patients also tend to be older/sicker)
- Instrument: postcode district of the treatment site (patients assigned to a site are not perfectly self-selected)
- Method: Two-Stage Least Squares
- Output: Unconfounded causal estimate of distance on no-show probability

**9c. Double Machine Learning (DML):**
- Estimates the average treatment effect (ATE) of an intervention while flexibly controlling for many confounders via cross-fitting
- Method: Neyman-orthogonal scores with K-fold nuisance estimation (scikit-learn cross-val + XGBoost for nuisance models)
- Output: ATE with standard error and 95% confidence interval
- Example: "Sending a reminder reduces no-show probability by 0.06 (± 0.02) on average"

---

### Model 10: Event Impact Model (`ml/event_impact_model.py`)

**Purpose:** Predict how external events (extreme weather, major local incidents, public transport disruption, sporting events) will affect attendance and treatment duration.

**Method:**
- **NLP component**: VADER sentiment analysis + keyword detection on event descriptions (via RSS feeds and manual entry)
- **Distance-weighted decay**: events affect patients more if the event is geographically proximate to the patient's postcode
- **Impact matrix**: pre-calibrated effect sizes for each event type × severity combination

**Output per event:**
- Expected no-show rate increase (percentage points)
- Expected duration buffer needed (extra minutes per patient)
- Recommended operating mode (Normal / Elevated / Crisis / Emergency)

**Event types:** Weather (rain, snow, extreme heat), Traffic Incident (road closure, accidents), Health Alert (local outbreak), Infrastructure Failure (power/transport), Emergency, Social Event, Sports Event.

---

### Model 11: Conformal Prediction (`ml/conformal_prediction.py`)

**Purpose:** Provide prediction intervals with a mathematically guaranteed coverage probability — unlike most ML models which give intervals based on heuristics, conformal intervals are valid with finite-sample guarantees.

**Method:** Two variants:
- **Split Conformal**: held-out calibration set used to compute non-conformity scores; coverage guarantee ≥ 1−α for any α
- **Conformalized Quantile Regression (CQR)**: combines quantile regression with conformal calibration for adaptive-width intervals (wider for more uncertain cases)

**Two instances:**
- `conformal_duration`: guaranteed-coverage duration intervals (e.g., 90% interval)
- `conformal_noshow`: guaranteed-coverage no-show probability intervals

**Output:** E.g., "90% coverage interval for duration: [78, 134] minutes — this interval contains the true duration with ≥ 90% probability across the calibration set."

**Use case:** Regulatory and audit contexts where interval validity guarantees are needed; also for flagging when a prediction is operating outside the training distribution (very wide intervals = novel case).

---

### Model 12: MC Dropout (`ml/mc_dropout.py`)

**Purpose:** Bayesian uncertainty quantification using Monte Carlo Dropout — decompose total prediction uncertainty into epistemic (model uncertainty, reducible with more data) and aleatoric (inherent noise, irreducible).

**Method:** Standard neural network with dropout layers enabled at inference time (not just training). Running T=50 forward passes through the same input yields a distribution of predictions:
- **Mean**: best estimate
- **Variance from T passes**: total uncertainty
- **Epistemic uncertainty**: variance from model (high for novel/out-of-distribution inputs)
- **Aleatoric uncertainty**: residual variance after model uncertainty is removed

**Output:**
- Point prediction (mean of 50 passes)
- Total uncertainty (std dev of 50 passes)
- Epistemic uncertainty component
- Aleatoric uncertainty component

**Use case:** If epistemic uncertainty is high for a patient, the model is telling you it hasn't seen enough similar cases — flag for manual review. If aleatoric is high, no amount of additional data will reduce uncertainty — the patient is inherently unpredictable.

## 7. Optimisation Engine

The optimiser (`optimization/optimizer.py`) uses **Google OR-Tools CP-SAT** to assign patients to chairs and time slots, maximising a weighted combination of 6 objectives simultaneously.

### Objectives

| Objective | Weight (balanced) | Direction |
|-----------|------------------|-----------|
| Priority score | 0.30 | Maximise (P1 patients scheduled first) |
| Chair utilisation | 0.25 | Maximise (fill chairs efficiently) |
| No-show risk exposure | 0.15 | Minimise (prefer low-risk appointments early) |
| Patient waiting time | 0.15 | Minimise (prefer earlier appointment times) |
| Schedule robustness | 0.10 | Maximise (preserve buffer between appointments) |
| Patient travel | 0.05 | Minimise (prefer closer sites) |

### Constraints (always enforced)

- No two patients in the same chair at the same time
- Operating hours: 08:00–18:00 (extended to 20:00 in elevated/crisis mode)
- Chair capacity per site cannot be exceeded
- Patients requiring 1:1 nursing must have a trained nurse available
- Priority 1 (Critical) patients must be scheduled within 2 days
- Priority 2 (High) patients within 7 days

### Pareto Frontier

The **Pareto Frontier** function (`POST /api/optimize/pareto`) generates 5 solutions simultaneously, one per weight profile:

| Profile | Weights (priority / util / noshow / wait / robust / travel) |
|---------|-------------------------------------------------------------|
| Balanced | 0.30 / 0.25 / 0.15 / 0.15 / 0.10 / 0.05 |
| Max Throughput | 0.15 / 0.45 / 0.10 / 0.10 / 0.10 / 0.10 |
| Patient First | 0.40 / 0.10 / 0.10 / 0.25 / 0.05 / 0.10 |
| Risk Averse | 0.20 / 0.15 / 0.30 / 0.10 / 0.20 / 0.05 |
| Robust | 0.20 / 0.15 / 0.15 / 0.10 / 0.30 / 0.10 |

### Urgent Squeeze-In

The squeeze-in engine (`optimization/squeeze_in.py`) adds urgent patients to a published schedule without re-running the full optimiser.

Three strategies (auto-selected based on schedule state):
1. **Gap**: insert into empty time slots ≥ 15 minutes
2. **Double-booking**: book alongside an existing appointment where predicted no-show probability > 40%
3. **Rescheduling**: move lower-priority appointments to create a slot for a P1/P2 patient

## 8. Real-Time Monitoring

Three background monitors run continuously, aggregating into a single severity score used by the optimiser and event impact model.

| Monitor | Source | Update interval | API |
|---------|--------|----------------|-----|
| Weather | Open-Meteo (free, no key) | Every 60 min | `GET /api/events` |
| Traffic | TomTom API (key optional) | Every 5 min | `GET /api/events` |
| News / RSS | BBC Wales + Wales Online + Gov.uk alerts | Every 15 min | `GET /api/events` |

**Combined event severity score:**
```
severity = 0.3 × weather_severity + 0.4 × traffic_severity + 0.3 × news_severity
```

Score thresholds determine operating mode:
- 0.0–0.30: Normal
- 0.30–0.50: Elevated (+15% buffer on all appointments)
- 0.50–0.80: Crisis (+25% buffer; P4 patients rescheduled)
- > 0.80: Emergency (+50% buffer; P1–P2 only; community backup sites activate)

### Distributionally robust fairness certificates (§4.1)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/fairness/audit` | Legacy response — now augmented with `dro_certificate` block (worst-case gap, certified verdict, narrative) |
| GET  | `/api/fairness/dro/status` | Certifier config (ε, δ, confidence, hard-constraint mode) + run count + last narrative |
| POST | `/api/fairness/dro/certify` | On-demand Wasserstein-DRO certificate against the live schedule; body: `{group_by, epsilon, delta}` |
| GET  | `/api/fairness/dro/last` | Full per-pair detail of the most recent certificate |
| POST | `/api/fairness/dro/config` | Retune `epsilon`, `delta`, `confidence`, `enforce_as_hard_constraint` |

Upgrades fairness from a soft objective-function penalty (trade-off-able) to a hard constraint with a certificate.  For a binary outcome `Y = 1[scheduled]` and 1-Wasserstein ball of radius ε, the plug-in dual bound is `worst_case_gap = |empirical_gap| + ε · (1/π_1 + 1/π_2)`, and the operator requires `worst_case_gap ≤ δ` to certify.  Additional `se_adjusted_upper` layer applies a Wald z-score for finite-sample protection.  Invisible integration: **every `run_optimization()` attaches a certificate** to `app_state['optimization_results']['fairness_dro_certificate']`; when `enforce_as_hard_constraint=True`, a failed certificate raises a `fairness_blocked` flag that downstream consumers can refuse to commit.  JSONL log at `data_cache/dro_fairness/certificates.jsonl`.  **No UI panel.**

### Individual (Lipschitz) fairness (§4.2)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/fairness/audit` | Legacy + DRO response — now augmented with `lipschitz_certificate` block (violation rate, worst excess, top-K violating pairs, narrative) |
| GET  | `/api/fairness/lipschitz/status` | Certifier config (L, τ, budget, features, hard-constraint mode) + run count + last narrative |
| POST | `/api/fairness/lipschitz/certify` | On-demand Lipschitz certificate against the live schedule; body: `{L, tau}` |
| GET  | `/api/fairness/lipschitz/last` | Full top-K violating pairs from the most recent certificate |
| POST | `/api/fairness/lipschitz/config` | Retune `L`, `tau`, `violation_budget`, `features`, `enforce_as_hard_constraint` |

Dwork et al. (2012) individual-fairness guarantee: patients with close feature profiles (d ≤ τ) should receive close outcomes (|f(x_i) − f(x_j)| ≤ L·d(x_i, x_j)).  Detects "unfair-by-coincidence" cases where two effectively identical patients land on opposite sides of the scheduled / unscheduled boundary.  Features min-max scaled to [0, 1]; Euclidean distance divided by √d for interpretability.  `sklearn.neighbors.NearestNeighbors(radius=τ)` runs in near-linear time.  **Lazy-constraint emitter** (`iter_violating_pairs`) returns `(patient_a, patient_b, excess)` triples that CP-SAT can translate into clauses forbidding the current assignment — the mechanism §4.2 asks for.  Invisible integration: every `run_optimization()` attaches `lipschitz_certificate` + `lipschitz_blocked`.  JSONL log at `data_cache/individual_fairness/certificates.jsonl`.  **No UI panel.**

### Safety guardrails with runtime monitoring (§4.3)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET  | `/api/safety/status` | Enabled rules + last verdict + narrative + run counter |
| POST | `/api/safety/evaluate` | On-demand safety check against the current schedule |
| GET  | `/api/safety/last` | Full violation list from the most recent report |
| POST | `/api/safety/config` | Toggle rules, retune thresholds, set `enforce_as_hard_gate` |

Runtime monitor wrapping `run_optimization()`.  Implements the brief's "technically optimal but clinically dangerous" rejection pattern with eight built-in rules: `critical_slack_floor` (CRITICAL — priority-{1,2} patients with < 5 min slack), `chair_capacity` (CRITICAL), `concurrent_chair_overlap` (CRITICAL), `long_infusion_cutoff` (HIGH — long infusions past 13:00), `travel_time_ceiling` (HIGH), `wait_time_ceiling` (HIGH), `consecutive_high_noshow` (MODERATE — cascade risk), `double_book_density` (MODERATE).  Verdict: any CRITICAL → `reject`; any HIGH/MODERATE → `warn`; else `accept`.  Invisible integration: **every `run_optimization()` attaches `safety_report` + `safety_blocked`**; when `enforce_as_hard_gate=True` (default), a `reject` verdict sets `safety_blocked=True` so downstream consumers refuse to commit.  Rule-registry is pluggable — new rules register at runtime via `monitor.register_rule`.  JSONL log at `data_cache/safety_guardrails/reports.jsonl`.  **No UI panel.**

### Counterfactual fairness audit (§4.4)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/fairness/audit` | Legacy + DRO + Lipschitz — now augmented with `counterfactual_certificate` block |
| GET  | `/api/fairness/counterfactual/status` | Auditor config (postcode, effect size, budget, predictor method) + run counter + last narrative |
| POST | `/api/fairness/counterfactual/audit` | On-demand audit; body: `{counterfactual_postcode, min_effect_size}` |
| GET  | `/api/fairness/counterfactual/last` | Full top-K flip cases from the most recent audit |
| POST | `/api/fairness/counterfactual/config` | Retune cf postcode, min effect size, flip budget, deprivation map, `flip_downstream` |

Tests for **proxy discrimination** the brief calls out: for every rejected patient, flip their postcode to an affluent counterfactual (default CF10) and ask whether a lightweight scheduleability predictor (logistic regression on age, priority, duration, deprivation, distance, travel-time, no-show) would now accept them.  Flip rule: `Pr_cf ≥ decision_threshold AND Pr_cf − Pr_orig ≥ 0.05`.  Certificate passes iff `flip_rate ≤ 0.10`.  Two modes: **label-only** (default; only deprivation label flips — purest proxy-discrimination test) and **circumstances-rich** (`flip_downstream=true`; also scales distance / travel / no-show — "what if they actually lived there?").  Built-in Welsh postcode → WIMD-decile lookup (CF10 affluent=2, CF44 deprived=9).  Invisible integration: **every `run_optimization()` attaches `counterfactual_certificate`**.  JSONL log at `data_cache/counterfactual_fairness/audits.jsonl`.  **No UI panel.**

### Auto-scaling optimizer with timeout guarantees (§5.1)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET  | `/api/optimize/autoscale/status` | Wrapper config (cascade budgets, parallel configs, early-stop gap, feature flags) + last winner + narrative |
| POST | `/api/optimize/autoscale/optimize` | Run full cascade + parallel race + greedy fallback on the current cohort; commits best schedule to app_state |
| GET  | `/api/optimize/autoscale/last` | Full per-attempt detail of the most recent run |
| POST | `/api/optimize/autoscale/config` | Retune `cascade_budgets`, `parallel_configs`, `parallel_time_limit_s`, `early_stop_gap`, `enable_{parallel,cascade,greedy_fallback}` |

Pure wrapper over the existing `ScheduleOptimizer`.  **Cascade:** tries budgets `[5s, 2s, 1s, 0.5s]` sequentially.  **4-way parallel race:** `{balanced, priority_heavy, throughput, robustness_heavy}` weight configurations run concurrently in a `ThreadPoolExecutor` (CP-SAT releases the GIL, so threads give real parallel speed-up); the best is chosen by `(-n_scheduled, solve_time, config_name)` for deterministic tie-breaking.  **Early stopping:** aborts when `|objective − best_bound| / |objective| ≤ 0.01` (1% optimality gap).  **Greedy fallback:** priority-first round-robin placement, 5-min slack, respects site capacity — runs in ms, always places at least the patients who fit.  Invisible integration: legacy `/api/optimize` keeps the single-solve path; **every `run_optimization()` attaches the most recent auto-scaling summary as `auto_scaling_last`** so downstream consumers (audit logs, dissertation analysis) have a canonical record of which path the wrapper chose.  JSONL log at `data_cache/auto_scaling/runs.jsonl`.  **No UI panel.**

### Human-in-the-loop override learning (§5.2)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET  | `/api/override/status` | Learner state: event count, model method, last narrative |
| POST | `/api/override/log` | Record one manual override (body: `OverrideEvent`) |
| POST | `/api/override/suggest` | Bulk-suggest across the current schedule |
| GET  | `/api/override/last` | Last 20 events + full status |
| POST | `/api/override/config` | Retune `suggest_threshold` (default 0.80), `min_events_for_fit`, `cold_start_prior`, `neighbourhood_hours` |

Closes the feedback loop the §5.2 brief describes: log every manual override, fit a logistic regression on `Pr(override | features)` where features = `(hour, dow, priority, duration, noshow_prob, site_idx)`, and preemptively suggest an alternative slot when `Pr(override) ≥ 0.80`.  Negative-class labels are synthesised from committed appointments that had no corresponding override event.  **sklearn logistic regression** is the primary model; **count-rate fallback** with Laplace smoothing handles sklearn-absent / degenerate-class cases.  **Cold-start prior** (0.10) applies before `min_events_for_fit` events have accumulated.  Suggestion rule: find the candidate alternative with the minimum predicted override probability; return `None` when no alternative improves on the original.  Invisible integration: **every `run_optimization()` attaches `override_suggestions`** so the post-schedule handoff to clinicians highlights likely-trouble slots up front.  JSONL logs at `data_cache/override_learning/{events,suggestions}.jsonl`.  **No UI panel.**

Reproduces the §5.3 brief's narrative pattern verbatim. ### Stochastic MPC scheduler (§5.5)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET  | `/api/mpc/status` | Controller config + arrival-model posterior + last decision |
| POST | `/api/mpc/decide` | On-demand MPC decision against the live schedule |
| POST | `/api/mpc/simulate` | Full-day rollout; body: `{"policies": ["greedy", "mpc"], "total_minutes": 480, "rng_seed": 42}` |
| POST | `/api/mpc/arrival/update` | Update Bayesian urgent-arrival model (Gamma-Poisson conjugate); body: `{"n_arrivals": 2, "minutes_observed": 5.0}` |
| GET  | `/api/mpc/arrival/rate` | Current posterior rate (per minute + per hour) |
| POST | `/api/mpc/config` | Retune `n_scenarios` (default 10), `lookahead_minutes` (default 120), `total_timeout_s` (default 0.5) |

Receding-horizon controller for real-time scheduling under three sources of uncertainty: urgent arrivals, late cancellations, and duration overruns.  **MDP formulation**: state = (time, chair statuses, queue, stats); action = per-chair assignment; reward combines completion credit, waiting cost, idle penalty, and a terminal penalty for unscheduled patients.  **Scenario rollout** samples K futures from the Poisson(λ) arrival model + per-patient (no-show, cancel) Bernoulli outcomes, then rolls each candidate action forward under a greedy fill and the terminal value function `V̂(S)`.  **Bayesian arrival model** (Gamma-Poisson conjugate) matches the §5.5 brief's `UrgentArrivalModel` verbatim: posterior mean = α/β; updates with `update(n_arrivals, minutes)`.  **Fail-safe**: on timeout > 500 ms the controller falls back to the squeeze-in handler or priority-first greedy, setting `used_fallback=True`.  **Invisible integration**: every `run_optimization()` attaches the last MPC decision to `app_state['optimization_results']['mpc_last_decision']` for audit trails.  JSONL logs at `data_cache/mpc_scheduler/{events,decisions,simulations}.jsonl`.  **No UI panel.**

## 12. Troubleshooting

**Server does not start / `ImportError`**
- Install missing packages: `pip install flask torch pymc lifelines econml scipy statsmodels mapie`
- Check Python version: `python --version` (must be ≥ 3.10)

**Startup takes > 10 minutes**
- The Hierarchical Bayesian model (PyMC MCMC) can be slow on first run with large datasets
- Reduce training data to < 1,000 rows for development: copy a subset to `datasets/sample_data/historical_appointments.xlsx`

**No-show model shows AUC ≈ 0.50**
- The model has insufficient training data (< 500 rows) or the `Attended_Status` column is missing
- Check: `GET /api/ml/model-cards` and look at `training_samples` for the no-show model
- Minimum: 500 records with a mix of attended and no-show outcomes

**Real data not detected**
- Verify `datasets/real_data/patients.xlsx` exists (exact filename and extension)
- Check file permissions (must be readable)
- Restart the server after placing files

**Traffic monitoring disabled**
- Set `TOMTOM_API_KEY` environment variable before starting the server
- Free tier: 2,500 requests/day (sufficient for production use at a single site)

**CWT download fails after June 2026**
- This is expected — CWT decommissions June 2026
- The NDRS Cancer Data Hub source activates automatically on 1 July 2026
- Check status: `GET /api/data/nhs/status` → look at `ndrs_cancer_data_hub.cwt_transition`

**SACT v4.0 data not triggering Level 3 retrain**
- Data must be placed in `datasets/nhs_open_data/sact_v4/` as a `.csv` file
- Full retrain only triggers from 1 August 2026 onwards (first_complete phase)
- Before August 2026, only Level 1–2 recalibration is triggered
- Manual trigger: `POST /api/ml/recalibration/run` with body `{"level": 3}`

## 13. Data Governance & Security

**No real patient data is included in this repository.** All data in `datasets/sample_data/` is synthetically generated and contains no real NHS Numbers, names, or clinical records.

**If deploying with real hospital data:**

| Requirement | Details |
|-------------|---------|
| Legal basis | NHS Data Security and Protection Toolkit (DSPT) Level 2+ |
| Data protection | UK GDPR Article 9 (special category health data) |
| Clinical governance | Caldicott Principles 1–8 |
| Access control | Restrict `datasets/real_data/` to authorised users only |
| Audit logging | Flask request logs capture all API calls; retain per local policy |
| Git exclusion | `datasets/real_data/*.xlsx` and `*.csv` are in `.gitignore` — do not override |
| Transmission | Use HTTPS (not HTTP) for any network-facing deployment |
| Retention | Apply trust data retention policies; delete or archive when no longer needed |

**PII fields in the system:**
- `NHS_Number` — 10-digit identifier
- `Person_Given_Name`, `Person_Family_Name`
- `Person_Birth_Date`
- `Patient_Postcode`

These fields are used only for schedule display and distance calculation. They are not transmitted to any external service.
