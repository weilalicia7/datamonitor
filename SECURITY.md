# Security Policy

## Supported Versions

This is an MSc dissertation project.  Only the `main` branch of this repository
is actively maintained.  No long-term support branches exist.

| Version | Supported          |
| ------- | ------------------ |
| `main`  | :white_check_mark: |
| any other tag / branch | :x: |

## Reporting a Vulnerability

If you discover a vulnerability in this codebase, please report it privately —
do **not** open a public GitHub issue that discloses the exploit path.

**Contact**: open a GitHub *private* vulnerability report on this repository
(Security tab → "Report a vulnerability").  For time-sensitive issues involving
real NHS patient data, contact the Cardiff University Information Governance
team at `informationgovernance@cardiff.ac.uk`.

Expected acknowledgement: within 7 calendar days.  Fix window: 30 days for
non-critical, 7 days for critical (remote code execution, data leak, auth
bypass).

## Threat Model

**What this repository IS**

- A research prototype for intelligent scheduling of Systemic Anti-Cancer
  Therapy (SACT) appointments, built as an MSc dissertation at Cardiff
  University.
- Designed for use with **synthetic data** generated by
  `datasets/generate_sample_data.py`.  All committed sample data uses
  `P#####` style patient IDs and fictional Welsh names.

**What this repository IS NOT**

- A production clinical system.
- An approved EHR or Patient Administration System (PAS).
- A place to store or process real NHS patient data without an explicit
  Data Sharing Agreement with the relevant NHS trust.

**Known limitations**

- No authentication or authorisation on the Flask endpoints (Tier 4.1 of the
  production-readiness plan).  See `docs/PRODUCTION_READINESS_PLAN.md`.
- No rate limiting.  Compute-heavy endpoints (`/api/optimize`,
  `/api/mpc/simulate`, `/api/twin/evaluate`) can exhaust the host if
  unthrottled.
- The scheduler unpickles its own trained models from disk on boot.  If an
  attacker can write to `models/` or `data_cache/feature_store/`, arbitrary
  code execution is possible via pickle deserialisation.  This is tracked as
  Tier 2.3 of the plan (SHA256-verified `joblib.load()`).

**Data-protection note**

If Channel 2 (real hospital data file-drop) is ever activated, every
event-log (`*.jsonl`), trained model (`*.pkl`), and Flask access log will
contain real NHS patient identifiers.  The `.gitignore` in this repository
blocks all of these from accidental commit; the `scripts/pre-commit.sh`
hook enforces the same block at commit time.  Do **not** disable either
safeguard on a checkout connected to real data.

## Scope

In scope:

- The code and documentation in this repository.
- The behaviour of the Flask application when run locally or in a container
  built from the included `Dockerfile`.

Out of scope:

- The trained ML models themselves (research artefacts).
- The synthetic sample data (by definition non-sensitive).
- Third-party dependencies (report upstream).
- Any deployment not under our control.
